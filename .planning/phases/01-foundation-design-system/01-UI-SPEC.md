## Spacing Scale

Declared values (must be multiples of 4), per Tailwind's default spacing scale — no project-specific override needed or requested in CONTEXT.md/RESEARCH.md:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, badge padding |
| sm | 8px | Compact element spacing, chip/tag padding |
| md | 16px | Default element spacing, card interior padding |
| lg | 24px | Section padding, screen edge padding on mobile |
| xl | 32px | Layout gaps between major screen regions |
| 2xl | 48px | Major section breaks (e.g. title block to CTA block on splash) |
| 3xl | 64px | Page-level spacing (rare — large empty states only) |

Exceptions: **44px minimum touch target** for any tappable element (buttons, toggle, card slots) — mobile-first PWA constraint from `.claude/CLAUDE.md` ("must work on phones," "smooth 60fps... mid-range phones"). 44px is not on the 4px-multiple ladder exactly (44 = 4×11, so it IS a valid multiple of 4 — no exception needed, confirmed compliant). Multi-touch player indicator dots (Phase 2, MTCH-02) may go smaller than 44px since they are not discrete tap targets — flagged for Phase 2's own UI-SPEC to re-confirm, not decided here.

This phase does not render any spacing-consuming UI itself (screens are placeholder headings, D-07) — this scale is the frozen contract Phases 2–4 inherit. No `@theme` spacing override is needed since Tailwind v4's default spacing scale already matches this table exactly.

## Typography

No custom font is loaded this phase (see Design System > Font). Sizes below use Tailwind's default type scale, which already lands on the required 3–4 role / 2-weight declaration:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px (`text-base`) | 400 (regular) | 1.5 |
| Label | 14px (`text-sm`) | 400 (regular) | 1.5 |
| Heading | 24px (`text-2xl`) | 600 (semibold) | 1.2 |
| Display | 40px (`text-4xl`) | 600 (semibold) | 1.1 |

Rationale: `Screens.png` shows a clear two-weight system — body/label copy is a lighter weight, while screen titles ("Setup Screen," "Finger Selection," result headlines) and the neon "TRUTH / DARE" wordmark are visually bold/heavy. Two weights (400/600) is enough to express that hierarchy without introducing a third weight the phase doesn't need. Display (40px) is reserved for the splash wordmark treatment (Phase 4, UX-01) and large numeric displays (e.g. countdown timer, Phase 4 UX-08) — not used by this phase's placeholder screens.

Placeholder screen headings (D-07, this phase's only rendered text) should use the **Heading** role (`text-2xl font-semibold`, 24px/600/1.2) on `text-primary` — establishes the pattern early for the 7 stub files without overcommitting to Display-scale treatment that belongs to Phase 4's real splash screen.

## Color

| Secondary (30%) — `surface` | `#1C0E2E` / `oklch(0.20 0.06 301)` | Card/panel glassmorphism surfaces, setup tiles, modal backgrounds (sampled from voting/setup panel fills) |
| Primary (10% family) — `primary` | `#8B2FE2` / `oklch(0.54 0.25 301)` | Neon purple — "Random" choice button, primary CTA gradient start, selected-tile highlight border |
| Primary (10% family) — `secondary` | `#E02B96` / `oklch(0.61 0.23 350)` | Neon pink — splash CTA button gradient start, "DARE" wordmark glow |
| Primary (10% family) — `accent` | `#40A1E9` / `oklch(0.68 0.14 245)` | Electric blue — "Truth" choice button border/fill, "TRUTH" wordmark glow |
| Primary (10% family) — `highlight` | `#F4CC50` / `oklch(0.86 0.15 91)` | Gold/orange — coin-flip circle, crown icon, "or" divider text, star/premium markers |
| Destructive — `failure` | `#E9243D` / `oklch(0.60 0.23 22)` | Fail-vote icon, failure result screen only |

Accent reserved for: the "10%" role is intentionally split across four named neon hues (`primary`/`secondary`/`accent`/`highlight`) rather than one single accent, because that is what the reference UI kit actually does — each hue is scoped to a *specific semantic role*, not general decoration:
- `primary` (purple): the "Random" truth-or-dare choice, primary gradient CTA starting color, selected/active state borders (e.g. selected pack tile in Setup)
- `secondary` (pink): primary CTA gradient starting color on the splash screen, "Dare" semantic color family root
- `accent` (blue): "Truth" semantic color, informational/neutral choice buttons
- `highlight` (gold): rewards, premium/selected markers, crown/coin iconography, the "+50"/"+10" score copy
- None of these four are ever used as plain body-text color or default button fill — each appears only on the specific interactive elements named above, matching PROJECT.md's color-role table exactly.

### Sourcing Method (per D-01, for traceability)

Values were extracted by cropping `public/images/Screens.png` (24-screen kit, 1181×1331px) into individual per-screen tiles at 2x scale and sampling solid-fill / border pixels with a Python/PIL script — not eyeballed or approximated from memory:

| Token | Source screen(s) | Sampled pixel evidence |
|-------|------------------|------------------------|
| `background` | Screen 1 (Splash), Screen 13 (Card Selection) | Multiple interior background samples: `#0E0025`, `#1C0130`, `#04000F`, `#04000A` — averaged/rounded to `#0A0414` |
| `surface` | Screen 6 (Setup), Screen 17 (Player Voting) | Card panel fill sampled at `#1C0E2E`-equivalent darkness, one step lighter than background |
| `primary` | Screen 10 (Choose Challenge Type) "Random" button border, Screen 6 Setup CTA left edge | Border peak `rgb(141,23,177)`; CTA gradient left `rgb(113,46,227)` — averaged to `#8B2FE2` |
| `secondary` | Screen 1 Splash CTA button left edge, "DARE" wordmark letterforms | CTA left `rgb(224,43,150)` / `#E02B96`; wordmark saturated sample `#E851D4` — CTA value used as canonical (button fill is more representative than glow-blurred text) |
| `accent` / `truth` | Screen 10 "Truth" button border, "TRUTH" wordmark | Border peak `rgb(64,161,233)` / `#40A1E9`; wordmark saturated sample `#5BCBFC` — border value used as canonical |
| `highlight` | Screen 1 "or" divider text, Screen 6 crown icon | "or" glow-core `#FEF364`; crown icon fill `#DDA242` — averaged/rounded to `#F4CC50` |
| `dare` | Screen 10 "Dare" button border | Border peak `rgb(210,36,97)` / `#D22461` |
| `success` | Screen 17 Player Voting, green check-circle | Ring sample `rgb(80,191,58)` / `#50BF3A` |
| `failure` | Screen 17 Player Voting, red X-circle | Ring sample `rgb(233,36,61)` / `#E9243D` |
| `premium` | Screen 6 Setup, crown icon | Icon fill `rgb(221,162,66)` / `#DDA242` |

All RGB→OKLCH conversions performed via the standard sRGB→linear→OKLab→OKLCH transform (not approximated) so the `@theme` tokens can be declared directly in OKLCH per D-02's Tailwind v4 CSS-native token format.

### Glow Utility Tokens (per D-03)

Reusable `--shadow-*` tokens in the same `@theme` block, generating `shadow-glow-*` utilities. Glow color = the token's own hue at reduced opacity, consistent with the neon-outline treatment visible on every button/badge in `Screens.png`:

| Token | Value | Usage |
|-------|-------|-------|
| `--shadow-glow-primary` | `0 0 24px 4px oklch(0.54 0.25 301 / 0.55)` | Purple glow — Random button, selected tiles, primary CTA |
| `--shadow-glow-secondary` | `0 0 24px 4px oklch(0.61 0.23 350 / 0.55)` | Pink glow — splash CTA, Dare-family accents |
| `--shadow-glow-accent` | `0 0 24px 4px oklch(0.68 0.14 245 / 0.55)` | Blue glow — Truth button, informational badges |
| `--shadow-glow-highlight` | `0 0 24px 4px oklch(0.86 0.15 91 / 0.55)` | Gold glow — crown, coin, premium markers |
| `--shadow-glow-success` | `0 0 20px 3px oklch(0.71 0.20 141 / 0.5)` | Green glow — pass/success states only |
| `--shadow-glow-failure` | `0 0 20px 3px oklch(0.60 0.23 22 / 0.5)` | Red glow — fail states only |

Glow radius/spread (24px blur, 4px spread for primary hues; 20px/3px for the narrower-use success/failure hues) is Claude's discretion per CONTEXT.md — no literal blur value was specified upstream. Phases 2–4 must consume these named utilities (`shadow-glow-primary`, etc.) rather than writing inline `box-shadow` values, per D-03's explicit anti-pattern warning in RESEARCH.md.

## Copywriting Contract

This phase ships no gameplay copy (placeholder screens only, D-07) — the contract below defines the **pattern** for placeholder content this phase, and flags what real copy will need in Phases 2–4.

| Element | Copy |
|---------|------|
| Placeholder screen heading | Lowercase phase-name string per screen (matches RESEARCH.md's code example pattern), e.g. `"start"`, `"setup"`, `"touchSelection"` — a dev-facing label confirming the router wired the correct screen, not user-facing copy. Rationale: D-07 explicitly scopes placeholder content to Claude's discretion, and RESEARCH.md's own example (`<h1>start</h1>`) establishes this minimal convention; keeping it literal to `state.phase` string values gives the 4 parallel devs an unambiguous "which screen am I looking at" signal during Phase 2-4 development. |
| PWA install prompt (browser-native, not app copy) | Sourced from manifest `name`/`short_name` per D-11 — "Truth or Dare — Finger Roulette" / "Truth or Dare". No custom install-prompt UI copy is written this phase (browser-native prompt only). |
| Empty state heading | Not applicable this phase — no data-driven screens exist yet (card grid, player list, etc. are Phase 2/3 scope) |
| Empty state body | Not applicable this phase |
| Error state | Not applicable this phase — only error-handling surface in Phase 1 is the localStorage `JSON.parse` try/catch (Pattern 3, RESEARCH.md), which fails silently to `defaultSettings` with no user-visible error copy (matches Security Domain guidance: malformed localStorage should not crash or alarm the user, just reset silently) |
| Destructive confirmation | Not applicable this phase — no destructive actions exist yet (Restart/Change Settings copy is Phase 2 UX-06 scope, next-round flow) |

**Forward note for Phase 2/3/4 UI-SPECs:** when those phases replace these placeholders with real screens, the primary CTA labels, empty-state copy, and destructive-action confirmations (e.g. "Restart Game" on `NextRoundScreen`) must be defined in *their own* UI-SPEC.md — this phase intentionally leaves them out of scope per the phase boundary in CONTEXT.md.

## Implementation Reference: `@theme` Block

For direct use by the planner/executor — the full token set this UI-SPEC resolves to, ready to drop into `src/index.css` per RESEARCH.md Pattern 1 (values below are FINAL, replacing RESEARCH.md's placeholder OKLCH figures per Open Question 1):
